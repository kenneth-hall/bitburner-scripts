# Refactor spec: Formulas.exe math with legacy fallback (Phase 4)

## Context

Work in `C:\Users\admin\bitburner-scripts` (edit `src/`; `npm run dev` runs the viteburner watcher that syncs into the game). Phase 3 (see `phase-03-batcher-refactor.md`) added the pipeline reservation waterfall, efficiency-score ranking, and shrink gating. The exported log confirms it works: over a 12.7-minute window, all 408 launched batches ran at full `HACK_FRACTION` with zero shrinks and zero failed launches, utilization averaged 88%, and skips behaved as saturation rather than fraction decay.

The same log surfaced the problem Phase 4 attacks alongside the Formulas upgrade: **the batch target flipped 14 times in 12.7 minutes** despite `RANK_HYSTERESIS`. Root cause is the known Phase 3 distortion — scores are sampled at *current* security, so a target's score jumps as prep completes and craters when it drifts. During fast early-run leveling, challengers routinely clear the 1.25× bar on sampling noise alone, and every flip abandons the incumbent pipeline's commitment.

Formulas.exe (`ns.formulas`, owned — purchased from the darknet) computes exact values for *hypothetical* server/player states, which removes that distortion and several others Phase 3 could only document in comments. But **Formulas.exe is a purchase, and resets wipe purchases.** Phase 4's core requirement: the formulas math is an upgrade layered on top of the existing calculations, never a replacement. Every current calculation must remain intact and automatically in use whenever Formulas.exe is absent.

## Structure: one module owns the branch

New file `src/sampling.js` — the single place where "formulas or legacy?" is decided per calculation. `daemon.js` and `targets.js` call its functions and never branch on mode themselves (they receive/pass a `useFormulas` flag or the module reads it per call — pick one seam and keep it consistent). This module is `ns`-dependent by design; the purity rule protects `scheduler.js` only.

Move into it, unchanged in behavior:

- `sampleBatchFields(ns, target, hackFraction)` from `daemon.js` — becomes the legacy branch of the new `sampleBatchFields(ns, target, hackFraction, useFormulas)`. The return shape gains one field on both branches — `steadyWeakenTime`, the cost-basis duration (see batch sizing); the legacy branch just copies `weakenTime` into it, so legacy behavior is unchanged.
- `samplePrepFields(ns, hosts, target)` and its `countInFlightThreads` helper from `daemon.js` — same treatment.
- The steady-state thread plan + score inputs currently inline in `targets.js` (`hackThreads`/`growThreads`/`weakenThreads`/`weakenTime`/`hackAnalyzeChance` sampling) — exposed as something like `steadyStatePlan(ns, server, useFormulas)`. `targets.js` keeps eligibility filtering, sorting, and display.

Weaken math needs no formulas branch anywhere: `ns.weakenAnalyze` is exact and state-independent (`formulas.hacking.weakenEffect` is the same number at 1 core, per its doc). Keep the existing weaken sizing on both paths.

## The formulas branch, calculation by calculation

All formulas functions take `(server, player)` objects: `ns.getServer(hostname)` and `ns.getPlayer()` (2 GB and 0.5 GB static — see RAM note in acceptance). To evaluate a hypothetical state, copy the real server object and override fields (e.g. `{ ...ns.getServer(host), hackDifficulty: minDifficulty, moneyAvailable: moneyMax }` — exact field names from the docs). `ns.formulas.mockServer()` exists but a real-server copy carries all the growth/level fields correctly; prefer the copy.

**Batch sizing (`sampleBatchFields`, formulas branch).** Thread counts and cost basis are evaluated at the *prepped* state — min security, max money — regardless of the target's current state. Durations are **not** (see below):

- hack threads: `ceil(hackFraction / formulas.hacking.hackPercent(preppedServer, player))`.
- hack security bump (sizes weaken1): `ns.hackAnalyzeSecurity(hackThreads)` — **without** the host argument on this branch. Per its doc, passing a host caps the threads by the server's state; "it's thread-math not state-math" only holds with the arg omitted. The legacy branch keeps passing `target.server` verbatim (behavior frozen).
- grow threads: simulate the state at the moment grow *lands* — the `growThreads` doc is explicit that what matters is the state when the grow **finishes**, not when it starts. At grow's landing, weaken1 has already landed (H → W1 → G → W2), so security is `minDifficulty`, not min + the hack bump. Money is `moneyMax × (1 − hackThreads × hackPercent)` — the *ceil'd* thread count, not the raw fraction, since the overshoot steals slightly more. **Clamp it (review finding): `postHackMoney = Math.max(1, moneyMax * (1 - hackThreads * hackPercent))`** — when per-thread steal is large (weak target, high level), `ceil` can push `hackThreads × hackPercent` to ~1 and the product to ~$0, and the doc is silent on zero starting money; same defensive style as the prep floor below. So: `formulas.hacking.growThreads({ ...preppedServer, moneyAvailable: postHackMoney }, player, moneyMax)`. Apply `GROW_BUFFER` to the result exactly as the legacy branch does, *before* sizing weaken2 off it. Two accuracy caveats to record as comments at the sizing site, no behavior change: **(i)** this math is exact for server/player state at **1 core** — host-core effects (e.g. home with upgraded cores) aren't modeled, and over-sizing grow/weaken is the safe direction; **(ii)** `isPrepped` admits launches down to `DRIFT_MONEY_FRACTION` (90%) money, where max-money sizing undersizes grow slightly — `GROW_BUFFER` covers that band too (one more reason it stays untouched per the ground rules).
- durations for batch timing: **current-state `getHackTime`/`getGrowTime`/`getWeakenTime` on both branches — the formulas branch does not touch these.** A job's real duration is fixed by the server's state when it starts, and `isPrepped` tolerates launches up to min+`DRIFT_SEC_EPSILON` security. All three durations scale by the same factor with security (grow/weaken are fixed multiples of hack time), so planning at exactly-min while launching at min+ε shifts each landing by ε-factor × *its own duration* — weaken's shift is 4× hack's, which reaches seconds against 200ms spacing, and weaken2 can land before grow. Legacy's same-tick sampling is exec-time-accurate; keep it.
- `steadyWeakenTime` (new field, cost basis only): the formulas branch sets it to `formulas.hacking.weakenTime(preppedServer, player)`; the legacy branch copies `weakenTime`. `pipelineDepth` and the reserve's `fullBatchRamCost` math consume this field, never the timing durations.
- hack chance (for `expectedSteal` display and ranking): `formulas.hacking.hackChance(preppedServer, player)`.

Why prepped-state for threads and cost: batches only launch against prepped targets, so the thread plan matches launch reality — but the *cost basis* (`fullBatchRamCost`, `pipelineDepth`) is sampled every tick **including during drift**, where the legacy branch overestimates hack threads, inflates weakenTime, and balloons the reserve (Phase 3 documented this as "safe direction"). The formulas branch makes the cost basis exact for the pipeline that will actually exist post-re-prep. Why *not* prepped-state for timing: the durations feed `additionalMsec` for real launches, where "exact" means exact for the state at exec — the current state. The two goals split cleanly across the two field sets. Update the comments at the call site: the overestimate caveat is legacy-only now — keep comments accurate per branch.

**Prep sizing (`samplePrepFields`, formulas branch).** Replace the grow-multiplier estimate with `formulas.hacking.growThreads(currentServer, player, moneyMax)` — exact inverse from actual current money. The `max(currentMoney, 1)` floor stays on **both** branches (resolved: the doc's linear-*and*-exponential remark implies the additive $1/thread term is modeled, but it is silent on zero starting money specifically, and the floor costs nothing — apply it to the copied server object's `moneyAvailable` on the formulas branch). The in-flight discounting logic is orthogonal to the math and stays identical on both branches.

**Ranking (`targets.js` via `steadyStatePlan`, formulas branch).** This is the churn fix. Score every target at its prepped state: `hackChance` at min security, `weakenTime` at min security, thread plan at prepped state. The score formula itself (`maxMoney × HACK_FRACTION × chance / (ramCost × weakenTimeSeconds)`) is unchanged — only its inputs stop depending on the target's current condition. Result: a target's score no longer moves as prep progresses or drift happens, so hysteresis only sees *real* changes (level-ups shifting times/chance, new targets unlocking). The legacy branch keeps current-state scoring and its pessimism comment verbatim. `RANK_HYSTERESIS` stays 1.25 — observe the flip rate before touching it.

One side effect prepped-state scoring introduces, and its fix: legacy's pessimism was accidentally a *readiness* gate — a newly unlocked target scores low until the waterfall preps it, so flips land when the challenger is warm. Scored at its prepped state, a fresh unlock gets its full score instantly and can clear 1.25× while stone-cold, flipping stage 1 into a full prep cycle during which nothing batches — trading Phase 3's noise-flips for cold-flips. So: `pickBatchTarget(targets, incumbentServer, hysteresis, challengerPrepped = true)` — a flip now also requires the challenger to be prepped. In formulas mode the daemon passes `isPrepped(liveTargetState(ns, targets[0]))`, sampled only when the top target isn't the incumbent (two 0.1 GB calls already referenced); in legacy mode it passes `true`, preserving today's behavior exactly per the ground rules. The incumbent-vanished case is untouched — `targets[0]` wins regardless, there's nothing to keep. The waterfall preps the challenger in the background either way; the flip just waits until it's warm.

`ns.getPlayer()` is sampled fresh per `getTargets`/per-tick sampling call, not cached across ticks — hacking level movement is precisely what the scores must track.

## Daemon hardening: the rename window

Verified this session: **in-place upgrades are already caught** — `refreshFreeRam` re-reads `getServerMaxRam` live, so `ns.cloud.upgradeServer` RAM appears within one `BATCH_INTERVAL_MS` tick, and new purchases land via `getHosts` within `CYCLE_MS`. The gap is **renames**: `fleetupgrade.js` renames every upgraded server (the name embeds the size), so for up to `CYCLE_MS` the daemon's cached host list holds hostnames that no longer exist, and the per-tick calls that take a hostname (`getServerMaxRam`, `getServerUsedRam`, `ns.ps`, `ns.scp`) presumably throw on a dead name — the docs don't spell out invalid-host behavior, so treat a throw as the working assumption and don't leave it reachable.

- Guard with `ns.serverExists(hostname)` (0.1 GB — verify in `markdown\bitburner.ns.serverexists.md`): at the top of each tick, drop vanished hostnames from the working host list before any per-host call, `tprintTs` an INFO once per disappearance, and — if anything was dropped — run `refreshCycle()` **immediately** instead of waiting out `CYCLE_MS`. The wait isn't cosmetic: while renamed hosts are missing from the list, their workers are invisible to `countBatchesInFlight`/`sumInFlightRam`, so `batchesInFlightBeforeTick` can read 0 with a full pipeline in flight — which opens the shrink gate and lets a runt launch mid-pipeline, the exact ratchet Phase 3 closed (the reserve also balloons off the same undercount). An immediate rescan closes that window within the tick.
- The guard is airtight, not best-effort, and a comment at the guard should say why: `fleetupgrade.js` contains no `await`, so it runs atomically between daemon ticks — every rename lands while the daemon sleeps — and the daemon's own tick has no awaits between the guard and its per-host calls. No try/catch needed.
- `fleetupgrade.js`: it currently ignores `renameServer`'s boolean and then calls `getServerMaxRam(newName)` in its summary — if a rename fails (the docs don't say whether it can while scripts are running on the box; assume it can fail), that summary line throws on a nonexistent name. Check the return value; on failure, warn, keep reporting under the old name. ## Logging additions (decided: skips + flips)

The exported log currently records only launch ticks, which made two things unverifiable offline this session: skip character (saturated vs empty-pipeline — the exact signal Phase 3 said to watch) and target flips (reconstructable only by diffing consecutive entries). Add both as first-class records, plus mode changes since they're nearly free:

- Every record (including existing batch entries) gains an `event` field: `"batch" | "skip" | "flip" | "mode" | "xcheck"` (the original draft omitted `xcheck` from this union — it's an event type like any other).
- `skip`: timestamp/time, `batchTarget`, `saturated` (boolean — the existing `batchSkipSaturated`), `batchesInFlight`, the `pipeline` block, `utilizationPct`. **Coalesce runs:** consecutive skip ticks with the same target and classification update the previous record in place (`count`, `firstTimestamp`, `lastTimestamp`) instead of appending — a long saturation stretch at 1-second cadence would otherwise evict hundreds of batch records from the ring buffer. **Flush lazily:** coalescing mutates the in-memory record only; rewrite the file when a record is *appended*, or at most every ~10 s — today the file is written only on launches, and the alternative is a full-buffer `JSON.stringify` at 1 Hz for the length of every saturation stretch.
- `flip`: timestamp/time, `from`, `to`, both scores, and the *outgoing* target's `commitmentPct` at the moment of the flip (the investment being abandoned — this is the number that makes churn cost visible). Implementation note: by flip-detection time the tick hasn't computed the old target's cost basis and never will again — cache the incumbent's `pipelineCost` and in-flight RAM from the previous tick rather than resampling a target on its way out. If the incumbent vanished from the target list entirely, record `fromScore: null`.
- `mode`: timestamp/time, `formulas` boolean, `forcedLegacy` boolean, and a `config` snapshot (`HACK_FRACTION`, `GROW_BUFFER`, `WEAKEN_BUFFER`, `DRIFT_SEC_EPSILON`, `DRIFT_MONEY_FRACTION`, `RANK_HYSTERESIS`, `BATCH_INTERVAL_MS`). One at startup too, so every log file states its mode *and is self-describing* — the log checker validates against the log's own recorded constants, not whatever the source tree says today.
- `xcheck`: written only on a cross-check mismatch (see automated verification below) — timestamp/time, target, and both branches' values for whatever disagreed.
- The ring buffer cap now means "last 1000 *events*". Note this in the file-level comment; anything reading the log must filter by `event`. Ordering subtlety: an in-place-updated skip's `lastTimestamp` can exceed the `timestamp` of records appended after it — record order is defined on `timestamp`/`firstTimestamp`, and the checker must treat it that way.

## Automated verification (tests + log checker)

The acceptance criteria below are split into *runnable* checks and *observed* checks. The runnable ones are code — write them as part of this phase and keep them green while implementing. Three layers:

**1. Offline unit tests (Node, no game).** Add `vitest` as a devDependency and a `"test"` script — with three setup traps handled up front: **(i)** version pairing: current vitest majors require vite 5/6; pin a vite-4-compatible vitest (0.34.x) unless a quick check shows a newer vitest's own nested vite leaves viteburner's vite 4 undisturbed — verify, don't assume. **(ii)** Vitest auto-loads `vite.config.ts` — which here boots the viteburner plugin — unless a dedicated `vitest.config.ts` exists; add one. **(iii)** That config must exclude `test/verify-log.test.js` from the default run (a separate script alone doesn't stop `vitest run` from globbing it), keeping `npm test` fast and game-independent. Tests live in `test/` at the repo root — **not** under `src/`: viteburner's watch pattern is `src/**`, so anything in `src/` gets pushed into the game. Two test files:

- `test/scheduler.test.js` — real assertions for the pure math that has only ever been eyeball-verified: `carveReservation` (largest-first order, hosts zeroed then skipped, over-reserve exhausts the pool, input not mutated), `pickBatchTarget` (hysteresis boundary: holds at just-under the factor, flips at just-over; incumbent missing from list; challenger gate: an over-threshold challenger with `challengerPrepped = false` does not flip, and the parameter's default `true` preserves old call sites), `planBatch` (landing offsets: every job's `additionalMsec + duration` equals `referenceTime + its landing offset`), `planPrep` (splitting across hosts, thread short-changing when RAM runs out, schedule's requested-vs-launched bookkeeping), `pipelineDepth`, `batchRamCost`, `isPrepped` edges.
- `test/sampling.test.js` — the legacy-parity net for the code move. Build a small mock `ns` object with canned, deterministic returns (`hackAnalyze` → fixed fraction, `growthAnalyze` → simple stub, etc. — the mock does *not* imitate game formulas, it just needs to be deterministic; parity means "same code path, same outputs," not "matches the game"). Migration sequence, each step verifiable: **(a)** export `sampleBatchFields`/`samplePrepFields`/`countInFlightThreads` from `daemon.js` where they live today and write tests capturing their outputs against the mock — golden values, computed by hand from the mock's canned numbers; **(b)** move the functions into `sampling.js`, retarget the imports — tests must stay green untouched; **(c)** add the formulas branch with its own tests (mock `ns.formulas.hacking` with distinct canned values so the tests prove the right branch ran, and assert the mock formulas functions are *never called* when `useFormulas` is false — that's the airtight-gate test).

ESM trap, called out because it will bite: the `src/` files use `import`/`export` in `.js` files with no `"type": "module"` in package.json. Vitest transforms these fine; plain `node` will not. So anything that imports from `src/` (including the log checker below) runs *through vitest*, not bare node. Don't add `"type": "module"` to package.json without checking it doesn't disturb viteburner.

**2. Log checker (codified acceptance criteria).** `test/verify-log.test.js`, run via a separate script (`"verify:log": "vitest run test/verify-log.test.js"`) so `npm test` stays fast and game-independent. Reads `logs/daemon-batch-log.json` (path overridable via env var) and checks. **Old-format logs (review finding):** the daemon rewrites the file wholesale from an in-memory buffer that starts empty, so mixed-format files can't exist — no dual-format support; if the file lacks the `event` field / startup `mode` event, fail fast with a clear "pre-Phase-4 log, re-export from a new session" message.

- *Hard assertions (fail the run):* timestamps monotonic per event stream (on `timestamp`, or `firstTimestamp` for coalesced skips — an updated skip's `lastTimestamp` legitimately exceeds later records'); running counters (`totalBatchesSkipped`, `totalBatchesShrunk`) never decrease; every event has a valid `event` type and the fields this spec assigns it; every `batch` event's `hackFraction` equals the `HACK_FRACTION` recorded in the log's own startup `mode` event (self-describing logs — a session run under different constants must not fail a newer checker, and this drops the import from `src/`) *unless* the pipeline was bootstrapping — a sub-fraction batch with evidence of a non-empty pipeline in the same record is the ratchet signature and a real failure; skip records' coalescing is well-formed (`count` ≥ 1, `firstTimestamp` ≤ `lastTimestamp`).
- *Soft reports (printed, never fail):* flip rate per hour with each flip's abandoned `commitmentPct` (baseline to beat: 14 flips / 12.7 min, logged pre-formulas); skip breakdown saturated-vs-empty; shrink total delta over the window; reserve behavior across drift windows (in formulas mode, `pipeline.reserveGb` on unprepped-target entries should sit near neighboring prepped entries rather than ballooning). These are phase-of-game dependent — early-run leveling churns legitimately — so they inform rather than gate.

**3. In-game cross-check (runtime canary).** Compare like-for-like at the *current* state, where both branches describe the same state by construction — not at the prepped state, where they only coincide if the target is *exactly* at min/max. (`isPrepped` tolerates min+`DRIFT_SEC_EPSILON` and 90% money; inside that band durations diverge by several percent and thread counts by dozens, so gating on `isPrepped` would make the zero-xcheck acceptance criterion unachievable noise.) Once per `CYCLE_MS` in formulas mode, against the batch target *as it currently is*: `formulas.hacking.hackTime/growTime/weakenTime(realServer, player)` vs the `getHackTime` family (relative tolerance ~0.1%); `formulas.hacking.hackPercent(realServer, player)` vs `ns.hackAnalyze`; `formulas.hacking.hackChance` vs `ns.hackAnalyzeChance` (same tolerance). When the target needs grow (money below `DRIFT_MONEY_FRACTION × maxMoney`), also `formulas.hacking.growThreads(realServer, player, maxMoney)` vs the legacy prep estimate, bounded loosely — flag past ~2×; the additive-bonus modeling legitimately differs, but a swapped `growThreads` parameter order lands far outside 2×. **The grow comparison is soft (review finding):** legacy `growthAnalyze` ignores the $1/thread additive bonus (per the existing comment in `samplePrepFields`), so on a deeply drained target the legitimate divergence can plausibly exceed 2× — a grow breach writes an `xcheck` event with `soft: true` and does **not** count against the zero-xcheck acceptance criterion until a real session's data says the 2× bound is quiet in practice. The time/percent/chance checks stay hard. On any breach: `tprintTs` a WARN and write an `xcheck` log event with both value sets. This validates exactly what can silently break — parameter order and `Server`/`Person` field names — on any tick, no gating. The one thing it doesn't exercise is the prepped-state mock construction itself; that's three field overrides, covered by the unit tests' formulas-branch goldens, not by runtime checks.
